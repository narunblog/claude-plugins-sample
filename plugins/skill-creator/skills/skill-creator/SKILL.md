# スキルクリエイター

新しいスキルを作成し、反復的に改善するためのスキル。

大まかに言えば、スキルの作成プロセスは以下のようになる:

- スキルに何をさせたいか、おおよそどのように実現するかを決める
- スキルのドラフトを書く
- いくつかのテストプロンプトを作成し、スキルにアクセスできるClaudeで実行する
- ユーザーが結果を定性的・定量的に評価するのを手伝う
  - バックグラウンドで実行中に、定量的な評価がまだなければドラフトする（既にある場合は、そのまま使うか、変更が必要と感じれば修正する）。ユーザーに説明する（既存のものがあれば、それを説明する）
  - `eval-viewer/generate_review.py` スクリプトを使って結果をユーザーに見せ、定量的なメトリクスも確認できるようにする
- ユーザーの評価からのフィードバックに基づいてスキルを書き直す（また、定量的ベンチマークから明らかな欠陥がある場合も同様）
- 満足するまで繰り返す
- テストセットを拡大して、より大きな規模で再度試す

このスキルを使う際のあなたの仕事は、ユーザーがこのプロセスのどの段階にいるかを把握し、その段階を進めるのを手伝うことだ。例えば、ユーザーが「Xのスキルを作りたい」と言うかもしれない。意図を絞り込み、ドラフトを書き、テストケースを書き、評価方法を決め、すべてのプロンプトを実行し、繰り返すのを手伝える。

一方、ユーザーが既にスキルのドラフトを持っている場合は、評価/反復のパートに直行できる。

もちろん、常に柔軟であるべきで、ユーザーが「たくさんの評価は不要、一緒にバイブスでやろう」と言えば、それに従えばいい。

そして、スキルが完成した後（ただし順番は柔軟）、スキルのdescription改善ツールを実行できる。これにはスキルのトリガーを最適化するための専用スクリプトがある。

いいね？いいね。

## ユーザーとのコミュニケーション

スキルクリエイターは、コーディング用語への親しみ度が大きく異なる人々に使われる可能性がある。最近の傾向として（つい最近始まったばかりだが）、Claudeのパワーに触発されて、配管工がターミナルを開いたり、親や祖父母が「npm のインストール方法」をググったりするようになっている。一方で、大多数のユーザーはかなりコンピュータに詳しい。

そのため、コンテキストの手がかりに注意してコミュニケーションの仕方を調整してほしい！デフォルトとしての目安:

- 「評価」「ベンチマーク」はぎりぎりだが、OK
- 「JSON」「アサーション」については、ユーザーがそれらを知っているという明確な手がかりがない限り、説明なしで使わないこと

迷った場合は簡潔に用語を説明してOKで、ユーザーが理解できるか不安なら短い定義を添えて構わない。

## スキルの作成

### 意図の把握

まずユーザーの意図を理解する。現在の会話にユーザーがキャプチャしたいワークフローが含まれているかもしれない（例：「これをスキルにして」）。その場合は、まず会話履歴から回答を抽出する — 使用されたツール、ステップの順序、ユーザーが行った修正、観察された入出力形式。ギャップがあればユーザーに補ってもらい、次のステップに進む前に確認を取る。

1. このスキルでClaudeに何ができるようにしたいか？
2. いつこのスキルがトリガーされるべきか？（どのようなユーザーのフレーズ/コンテキスト）
3. 期待される出力形式は？
4. スキルが正しく動作するかを検証するためのテストケースを設定すべきか？客観的に検証可能な出力（ファイル変換、データ抽出、コード生成、固定ワークフローステップ）を持つスキルはテストケースの恩恵を受ける。主観的な出力（文体、アート）を持つスキルはテストケースが不要な場合が多い。スキルの種類に応じた適切なデフォルトを提案するが、最終判断はユーザーに委ねる。

### インタビューとリサーチ

エッジケース、入出力形式、サンプルファイル、成功基準、依存関係について積極的に質問する。テストプロンプトの作成はこの部分が固まるまで待つ。

利用可能なMCPを確認する — リサーチに有用（ドキュメント検索、類似スキル探索、ベストプラクティス調査）であれば、サブエージェントが利用可能なら並列でリサーチし、そうでなければインラインで行う。ユーザーの負担を減らすためにコンテキストを準備しておく。

### SKILL.mdの作成

ユーザーインタビューに基づいて、以下のコンポーネントを記入する:

- **name**: スキルの識別子
- **description**: いつトリガーするか、何をするか。これが主要なトリガーメカニズムであり、スキルの機能と使用すべき具体的なコンテキストの両方を含める。「いつ使うか」の情報はすべてここに入れ、本文には入れない。注意: 現在Claudeはスキルを「トリガーしなさすぎる」傾向がある — 有用な場面でも使わない傾向がある。これに対処するため、スキルのdescriptionを少し「押し気味」にする。例えば、「内部のAnthropicデータを表示するシンプルで高速なダッシュボードの構築方法」ではなく、「内部のAnthropicデータを表示するシンプルで高速なダッシュボードの構築方法。ユーザーがダッシュボード、データ可視化、内部メトリクス、あるいは何らかの会社データの表示を求めた場合は、明示的に『ダッシュボード』と言わなくても、必ずこのスキルを使うこと。」のように書く。
- **compatibility**: 必要なツール、依存関係（オプション、めったに不要）
- **スキルの残りの部分 :)**

### スキル作成ガイド

#### スキルの構造

```
skill-name/
├── SKILL.md (必須)
│   ├── YAMLフロントマター (name, description 必須)
│   └── マークダウンの指示
└── バンドルリソース (オプション)
    ├── scripts/    - 決定的/反復的タスク用の実行可能コード
    ├── references/ - 必要に応じてコンテキストに読み込むドキュメント
    └── assets/     - 出力で使用するファイル（テンプレート、アイコン、フォント）
```

#### 段階的開示

スキルは3段階のローディングシステムを使用する:
1. **メタデータ** (name + description) - 常にコンテキスト内（約100語）
2. **SKILL.md本文** - スキルがトリガーされるたびにコンテキスト内（500行以下が理想）
3. **バンドルリソース** - 必要に応じて（無制限、スクリプトはロードせずに実行可能）

これらの語数は目安であり、必要に応じて超えて構わない。

**主要パターン:**
- SKILL.mdを500行以下に保つ。この制限に近づいている場合は、追加の階層レイヤーと、スキルを使うモデルが次にどこを参照すべきかの明確なポインタを加える。
- SKILL.mdからリファレンスファイルを明確に参照し、いつ読むべきかのガイダンスを含める
- 大きなリファレンスファイル（300行超）には目次を含める

**ドメイン整理**: スキルが複数のドメイン/フレームワークをサポートする場合、バリアントごとに整理する:
```
cloud-deploy/
├── SKILL.md (ワークフロー + 選択)
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```
Claudeは関連するリファレンスファイルのみを読む。

#### 意外性のない原則

言うまでもないが、スキルにはマルウェア、エクスプロイトコード、システムセキュリティを侵害する可能性のあるコンテンツを含めてはならない。スキルの内容は、説明した場合にユーザーを意図的に驚かせるものであってはならない。誤解を招くスキルや、不正アクセス、データの窃取、その他の悪意ある活動を促進するために設計されたスキルの作成には応じないこと。ただし「XYZとしてロールプレイする」のようなものはOK。

#### 記述パターン

指示には命令形を使うことを推奨する。

**出力形式の定義** - 以下のように書ける:
```markdown
## レポート構造
必ずこの正確なテンプレートを使用する:
# [タイトル]
## エグゼクティブサマリー
## 主な発見
## 推奨事項
```

**サンプルパターン** - サンプルを含めると有用。以下のような形式で（ただし「入力」と「出力」がサンプル内にある場合は少し変えたほうがよい）:
```markdown
## コミットメッセージ形式
**サンプル 1:**
Input: Added user authentication with JWT tokens
Output: feat(auth): implement JWT-based authentication
```

### 記述スタイル

なぜ重要かをモデルに説明することを、厳格なMUSTの乱用より優先する。心の理論を使い、スキルを汎用的にし、特定の例に狭くなりすぎないようにする。まずドラフトを書き、新鮮な目で見直して改善する。

### テストケース

スキルのドラフト作成後、2〜3個の現実的なテストプロンプトを考える — 実際のユーザーが本当に言いそうなもの。ユーザーに共有する:（この正確な言い回しでなくてよい）「テストしたいケースをいくつか考えました。これで問題ないですか、それとも追加したいですか？」そして実行する。

テストケースを `evals/evals.json` に保存する。まだアサーションは書かない — プロンプトだけ。次のステップで実行中にアサーションをドラフトする。

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "ユーザーのタスクプロンプト",
      "expected_output": "期待される結果の説明",
      "files": []
    }
  ]
}
```

完全なスキーマ（後で追加する `assertions` フィールドを含む）は `references/schemas.md` を参照。

## テストケースの実行と評価

このセクションは一連の連続シーケンスであり、途中で止めないこと。`/skill-test` やその他のテスティングスキルは使用しない。

結果はスキルディレクトリの兄弟として `<skill-name>-workspace/` に配置する。ワークスペース内はイテレーションごとに整理し（`iteration-1/`, `iteration-2/` など）、各テストケースにディレクトリを割り当てる（`eval-0/`, `eval-1/` など）。事前にすべてを作成する必要はない — 進行に応じてディレクトリを作成する。

### ステップ1: すべての実行（with-skillとbaseline）を同じターンで起動する

各テストケースについて、同じターンで2つのサブエージェントを起動する — 1つはスキルあり、1つはスキルなし。重要: with-skill実行を先に起動してからベースラインに戻るのではなく、すべてを一度に起動してほぼ同時に完了するようにする。

**スキルあり実行:**

```
このタスクを実行してください:
- スキルパス: <path-to-skill>
- タスク: <eval prompt>
- 入力ファイル: <eval files if any, or "none">
- 出力先: <workspace>/iteration-<N>/eval-<ID>/with_skill/outputs/
- 保存する出力: <ユーザーが気にするもの — 例: ".docxファイル", "最終CSV">
```

**ベースライン実行**（同じプロンプト、ただしベースラインはコンテキストによる）:
- **新しいスキルの作成**: スキルなし。同じプロンプト、スキルパスなし、`without_skill/outputs/` に保存。
- **既存スキルの改善**: 旧バージョン。編集前にスキルをスナップショット（`cp -r <skill-path> <workspace>/skill-snapshot/`）し、ベースラインのサブエージェントをスナップショットに向ける。`old_skill/outputs/` に保存。

各テストケースの `eval_metadata.json` を作成する（アサーションは今は空でよい）。各評価にはテスト内容を表す説明的な名前をつける — 単なる "eval-0" ではなく。ディレクトリにもこの名前を使う。このイテレーションで新しいまたは変更された評価プロンプトを使う場合は、各新しい評価ディレクトリにこれらのファイルを作成する — 前のイテレーションから引き継がれるとは限らない。

```json
{
  "eval_id": 0,
  "eval_name": "descriptive-name-here",
  "prompt": "ユーザーのタスクプロンプト",
  "assertions": []
}
```

### ステップ2: 実行中にアサーションをドラフトする

実行完了を待つだけでなく、この時間を有効に使える。各テストケースの定量的アサーションをドラフトし、ユーザーに説明する。既にアサーションが `evals/evals.json` にある場合は、レビューして何をチェックするか説明する。

良いアサーションは客観的に検証可能で、説明的な名前を持つ — ベンチマークビューアーで結果を一目見ただけで、各アサーションが何をチェックしているか即座にわかるように読めるべきだ。主観的なスキル（文体、デザイン品質）は定性的に評価するほうがよい — 人間の判断が必要なものに無理にアサーションを適用しないこと。

ドラフトしたアサーションで `eval_metadata.json` ファイルと `evals/evals.json` を更新する。また、ビューアーでユーザーが見るものを説明する — 定性的な出力と定量的なベンチマークの両方。

### ステップ3: 実行完了時にタイミングデータをキャプチャする

各サブエージェントタスクが完了すると、`total_tokens` と `duration_ms` を含む通知を受け取る。このデータを即座に実行ディレクトリの `timing.json` に保存する:

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

このデータをキャプチャできるのはこの時だけ — タスク通知を通じて届き、他の場所には永続化されない。バッチ処理しようとせず、到着するたびに処理する。

### ステップ4: 採点、集計、ビューアーの起動

すべての実行が完了したら:

1. **各実行を採点する** — `agents/grader.md` を読む採点サブエージェントを起動し（またはインラインで採点し）、出力に対して各アサーションを評価する。結果を各実行ディレクトリの `grading.json` に保存する。grading.jsonのexpectations配列は `text`、`passed`、`evidence` フィールドを使用する（`name`/`met`/`details` やその他のバリアントではない）— ビューアーはこれらの正確なフィールド名に依存する。プログラムでチェックできるアサーションについては、目視ではなくスクリプトを作成して実行する — スクリプトのほうが速く、信頼性が高く、イテレーション間で再利用できる。

2. **ベンチマークに集計する** — skill-creatorディレクトリから集計スクリプトを実行する:
   ```bash
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>
   ```
   これにより `benchmark.json` と `benchmark.md` が生成され、各構成のpass_rate、時間、トークンが平均値±標準偏差とデルタで表示される。benchmark.jsonを手動生成する場合は、ビューアーが期待する正確なスキーマについて `references/schemas.md` を参照。
各with_skillバージョンをそのベースラインの前に配置する。

3. **アナリストパスを行う** — ベンチマークデータを読み、集計統計が隠す可能性のあるパターンを表面化する。何を探すかについては `agents/analyzer.md`（「ベンチマーク結果の分析」セクション）を参照 — スキルの有無に関わらず常に合格するアサーション（非弁別的）、高分散の評価（不安定な可能性）、時間/トークンのトレードオフなど。

4. **ビューアーを起動する** — 定性的出力と定量的データの両方:
   ```bash
   nohup python <skill-creator-path>/eval-viewer/generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```
   イテレーション2以降は、`--previous-workspace <workspace>/iteration-<N-1>` も渡す。

   **Cowork / ヘッドレス環境:** `webbrowser.open()` が利用できない場合やディスプレイがない環境では、`--static <output_path>` でサーバーを起動する代わりにスタンドアロンHTMLファイルを書き出す。フィードバックはユーザーが「Submit All Reviews」をクリックすると `feedback.json` ファイルとしてダウンロードされる。ダウンロード後、`feedback.json` をワークスペースディレクトリにコピーして、次のイテレーションで使えるようにする。

注意: ビューアーの生成にはgenerate_review.pyを使用する。カスタムHTMLを書く必要はない。

5. **ユーザーに伝える** — 例えば:「ブラウザで結果を開きました。2つのタブがあります — 『Outputs』で各テストケースをクリックしてフィードバックを残せます。『Benchmark』で定量的な比較が見られます。終わったらここに戻ってお知らせください。」

### ビューアーでユーザーが見るもの

「Outputs」タブは一度に1つのテストケースを表示する:
- **プロンプト**: 与えられたタスク
- **出力**: スキルが生成したファイル、可能な限りインライン表示
- **前回の出力**（イテレーション2以降）: 前回のイテレーションの出力を折りたたみで表示
- **正式な採点**（採点が実行された場合）: アサーションの合格/不合格を折りたたみで表示
- **フィードバック**: 入力中に自動保存されるテキストボックス
- **前回のフィードバック**（イテレーション2以降）: 前回のコメント、テキストボックスの下に表示

「Benchmark」タブは統計サマリーを表示: 各構成のpass rate、タイミング、トークン使用量、評価ごとのブレイクダウンとアナリストの観察。

ナビゲーションはprev/nextボタンまたは矢印キー。完了したら「Submit All Reviews」をクリックし、すべてのフィードバックが `feedback.json` に保存される。

### ステップ5: フィードバックを読む

ユーザーが完了を告げたら、`feedback.json` を読む:

```json
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "グラフに軸ラベルがない", "timestamp": "..."},
    {"run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..."},
    {"run_id": "eval-2-with_skill", "feedback": "完璧、最高", "timestamp": "..."}
  ],
  "status": "complete"
}
```

空のフィードバックはユーザーが問題ないと思ったことを意味する。具体的な指摘があったテストケースの改善に集中する。

完了したらビューアーサーバーを停止する:

```bash
kill $VIEWER_PID 2>/dev/null
```

## スキルの改善

これがループの核心部分だ。テストケースを実行し、ユーザーが結果をレビューし、そのフィードバックに基づいてスキルをより良くする必要がある。

### 改善の考え方

1. **フィードバックから一般化する。** ここで起きている大局的なことは、百万回（文字通り、あるいはそれ以上かもしれない）多くの異なるプロンプトで使えるスキルを作ろうとしていることだ。ここでは、より速く進むためにほんの数例を何度も反復している。ユーザーはこれらの例を隅々まで知っていて、新しい出力を素早く評価できる。しかし、あなたとユーザーが共同開発しているスキルがそれらの例でしか動かないなら、役に立たない。細かい過学習的な変更や、圧迫的に制約的なMUSTを入れるのではなく、頑固な問題があれば、異なる比喩を使ったり、異なる作業パターンを推奨したりしてみる。試すコストは比較的低く、素晴らしいものに行き着くかもしれない。

2. **プロンプトをスリムに保つ。** 効果を上げていないものを削除する。トランスクリプトを必ず読む — 最終出力だけでなく — スキルがモデルに非生産的なことに大量の時間を費やさせているように見える場合は、その原因となっているスキルの部分を取り除いて何が起きるか試してみる。

3. **「なぜ」を説明する。** モデルに求めていることの**理由**をできるだけ説明する。今日のLLMは*賢い*。良い心の理論を持ち、良いハーネスが与えられれば、機械的な指示を超えて本当に成果を出せる。ユーザーからのフィードバックが簡潔だったりイライラしていたりしても、タスクを実際に理解し、ユーザーが書いた理由と内容を理解し、この理解を指示に反映させるよう努める。ALWAYSやNEVERを全大文字で書いている場合は黄色信号 — 可能なら、なぜそれが重要かの推論を説明するよう書き直す。より人間的で、パワフルで、効果的なアプローチだ。

4. **テストケース間で繰り返される作業を探す。** テスト実行のトランスクリプトを読み、サブエージェントがすべて独立して類似のヘルパースクリプトを書いたり、同じマルチステップアプローチを取ったりしていないか注意する。3つのテストケースすべてでサブエージェントが `create_docx.py` や `build_chart.py` を書いた場合、それはスキルがそのスクリプトをバンドルすべき強いシグナルだ。一度書いて `scripts/` に置き、スキルにそれを使うよう指示する。これにより、将来のすべての呼び出しで車輪の再発明を防げる。

このタスクはかなり重要で（年間数十億ドルの経済的価値を創出しようとしている！）、思考時間がボトルネックではない。時間をかけてよく考えること。ドラフトの修正を書いてから、新鮮な目で見直して改善することを推奨する。本当にユーザーの立場になって、彼らが何を望み必要としているかを理解するよう最善を尽くしてほしい。

### イテレーションループ

スキルを改善した後:

1. 改善をスキルに適用する
2. すべてのテストケースを新しい `iteration-<N+1>/` ディレクトリで再実行する（ベースライン実行を含む）。新しいスキルを作成している場合、ベースラインは常に `without_skill`（スキルなし）— イテレーション間で変わらない。既存のスキルを改善している場合は、ベースラインとして何が適切か判断する: ユーザーが持ち込んだ元のバージョンか、前のイテレーションか。
3. `--previous-workspace` で前のイテレーションを指定してレビューアーを起動する
4. ユーザーがレビューして完了を告げるのを待つ
5. 新しいフィードバックを読み、再度改善し、繰り返す

以下の条件で終了する:
- ユーザーが満足だと言う
- フィードバックがすべて空（すべて問題なし）
- 有意義な進歩が得られなくなった

## 高度な機能: ブラインド比較

2つのスキルバージョン間のより厳密な比較が必要な場合（例: ユーザーが「新バージョンは本当に良くなった？」と聞く場合）、ブラインド比較システムがある。詳細は `agents/comparator.md` と `agents/analyzer.md` を参照。基本的なアイデアは: 2つの出力をどちらがどちらか伝えずに独立したエージェントに渡し、品質を判断させる。そして勝者が勝った理由を分析する。

これはオプションであり、サブエージェントが必要で、ほとんどのユーザーには不要。人間のレビューループで通常は十分。

## Description最適化

SKILL.mdフロントマターのdescriptionフィールドは、Claudeがスキルを呼び出すかどうかを決定する主要なメカニズムだ。スキルの作成または改善後、トリガー精度を向上させるためにdescriptionの最適化を提案する。

### ステップ1: トリガー評価クエリの生成

20個の評価クエリを作成する — トリガーすべきものとすべきでないものの混合。JSONとして保存:

```json
[
  {"query": "ユーザープロンプト", "should_trigger": true},
  {"query": "別のプロンプト", "should_trigger": false}
]
```

クエリは現実的で、Claude CodeまたはClaude.aiのユーザーが実際に入力するようなものでなければならない。抽象的なリクエストではなく、具体的で詳細なリクエスト。例えば、ファイルパス、ユーザーの仕事や状況に関する個人的なコンテキスト、カラム名と値、会社名、URL。少しの背景説明。小文字だったり、略語やタイプミスやカジュアルな口調を含むものもあり。さまざまな長さを混ぜ、明確なものよりエッジケースに焦点を当てる（ユーザーが確認する機会がある）。

悪い例: `"このデータをフォーマットして"`, `"PDFからテキストを抽出して"`, `"チャートを作って"`

良い例: `"上司からxlsxファイルが送られてきたんだけど（ダウンロードに入ってて、ファイル名は 'Q4 sales final FINAL v2.xlsx' みたいなやつ）、利益率をパーセンテージで表示する列を追加してほしいんだって。売上はC列でコストはD列だったと思う"`

**トリガーすべきクエリ**（8〜10個）については、カバレッジを考える。同じ意図のさまざまな言い方が欲しい — フォーマルなものからカジュアルなものまで。ユーザーがスキルやファイルタイプを明示的に名前で指定しないが明らかにそれが必要なケースを含める。珍しいユースケースや、このスキルが他と競合するが勝つべきケースも含める。

**トリガーすべきでないクエリ**（8〜10個）については、最も価値があるのはニアミス — スキルとキーワードやコンセプトを共有するが実際には別のものが必要なクエリ。隣接ドメイン、ナイーブなキーワードマッチではトリガーするが実際にはすべきでない曖昧な表現、スキルの機能に触れるが別のツールのほうが適切なコンテキストでのクエリ。

避けるべきこと: トリガーすべきでないクエリを明らかに無関係にしないこと。PDFスキルの否定テストとしての「フィボナッチ関数を書いて」は簡単すぎる — 何もテストしていない。否定ケースは本当にトリッキーであるべきだ。

### ステップ2: ユーザーとのレビュー

HTMLテンプレートを使って評価セットをユーザーに提示しレビューしてもらう:

1. `assets/eval_review.html` からテンプレートを読む
2. プレースホルダーを置換する:
   - `__EVAL_DATA_PLACEHOLDER__` → 評価アイテムのJSON配列（引用符なし — JS変数代入として）
   - `__SKILL_NAME_PLACEHOLDER__` → スキルの名前
   - `__SKILL_DESCRIPTION_PLACEHOLDER__` → スキルの現在のdescription
3. 一時ファイルに書き出し（例: `/tmp/eval_review_<skill-name>.html`）、開く: `open /tmp/eval_review_<skill-name>.html`
4. ユーザーはクエリの編集、should-triggerの切り替え、エントリの追加/削除ができ、「評価セットをエクスポート」をクリックする
5. ファイルは `~/Downloads/eval_set.json` にダウンロードされる — 複数ある場合に備えて最新のものをチェック（例: `eval_set (1).json`）

このステップは重要 — 悪い評価クエリは悪いdescriptionにつながる。

### ステップ3: 最適化ループの実行

ユーザーに伝える:「少し時間がかかります — バックグラウンドで最適化ループを実行し、定期的にチェックします。」

評価セットをワークスペースに保存してから、バックグラウンドで実行する:

```bash
python -m scripts.run_loop \
  --eval-set <path-to-trigger-eval.json> \
  --skill-path <path-to-skill> \
  --model <model-id-powering-this-session> \
  --max-iterations 5 \
  --verbose
```

モデルIDにはシステムプロンプトのもの（現在のセッションを動かしているモデル）を使い、トリガーテストがユーザーが実際に体験するものと一致するようにする。

実行中は、定期的に出力を確認して、どのイテレーションにいるかやスコアの状況をユーザーに伝える。

これにより完全な最適化ループが自動的に処理される。評価セットを60%訓練と40%ホールドアウトテストに分割し、現在のdescriptionを評価（各クエリ3回実行で信頼性のあるトリガー率を取得）し、失敗したものに基づいてClaudeのextended thinkingで改善案を提案する。訓練とテストの両方で各新しいdescriptionを再評価し、最大5回反復する。完了すると、イテレーションごとの結果を表示するHTMLレポートをブラウザで開き、`best_description` を含むJSONを返す — 過学習を防ぐため、訓練スコアではなくテストスコアで選択される。

### スキルトリガーの仕組み

トリガーメカニズムを理解すると、より良い評価クエリの設計に役立つ。スキルはClaudeの `available_skills` リストにname + descriptionとともに表示され、Claudeはそのdescriptionに基づいてスキルを参照するかどうかを決定する。重要なのは、Claudeは自分だけで簡単に処理できるタスクにはスキルを参照しないことだ — 「このPDFを読んで」のような単純な一ステップクエリは、descriptionが完璧にマッチしてもスキルをトリガーしない可能性がある。Claudeが基本ツールで直接処理できるからだ。複雑な、マルチステップの、または専門的なクエリは、descriptionがマッチする場合に確実にスキルをトリガーする。

これは、評価クエリがClaudeがスキルを参照することで実際に恩恵を受ける程度に実質的であるべきことを意味する。「ファイルXを読んで」のような単純なクエリは不適切なテストケース — description品質に関係なくスキルをトリガーしない。

### ステップ4: 結果の適用

JSON出力の `best_description` を取得し、スキルのSKILL.mdフロントマターを更新する。ビフォー/アフターをユーザーに見せ、スコアを報告する。

### パッケージングと提示（`present_files` ツールが利用可能な場合のみ）

`present_files` ツールにアクセスできるか確認する。できない場合はこのステップをスキップする。できる場合は、スキルをパッケージングして.skillファイルをユーザーに提示する:

```bash
python -m scripts.package_skill <path/to/skill-folder>
```

パッケージング後、ユーザーにインストールできるよう結果の `.skill` ファイルパスを案内する。

## Claude.ai固有の指示

Claude.aiでは、コアワークフローは同じ（ドラフト → テスト → レビュー → 改善 → 繰り返し）だが、Claude.aiにはサブエージェントがないため、いくつかの仕組みが変わる。適応すべき点:

**テストケースの実行**: サブエージェントがないため並列実行はできない。各テストケースについて、スキルのSKILL.mdを読み、その指示に従ってテストプロンプトを自分で実行する。一度に1つずつ。これは独立したサブエージェントほど厳密ではない（スキルを書いた本人が実行するので完全なコンテキストがある）が、有用なサニティチェックであり、人間のレビューステップが補完する。ベースライン実行はスキップする — リクエストされたタスクを遂行するためにスキルを使うだけ。

**結果のレビュー**: ブラウザを開けない場合（例: Claude.aiのVMにディスプレイがない、リモートサーバー上）、ブラウザレビューアーは完全にスキップする。代わりに、会話内で結果を直接提示する。各テストケースのプロンプトと出力を表示する。ユーザーが見る必要のあるファイル出力（.docxや.xlsxなど）があれば、ファイルシステムに保存してその場所を伝え、ダウンロードして確認できるようにする。インラインでフィードバックを求める:「これはどうですか？何か変えたいことはありますか？」

**ベンチマーキング**: 定量的ベンチマークはスキップする — サブエージェントなしではベースライン比較が意味をなさない。ユーザーからの定性的フィードバックに集中する。

**イテレーションループ**: 以前と同じ — スキルを改善し、テストケースを再実行し、フィードバックを求める — ただしブラウザレビューアーは使わない。ファイルシステムがあれば、結果をイテレーションディレクトリに整理できる。

**Description最適化**: このセクションには `claude` CLIツール（具体的には `claude -p`）が必要で、Claude Codeでのみ利用可能。Claude.aiにいる場合はスキップする。

**ブラインド比較**: サブエージェントが必要。スキップする。

**パッケージング**: `package_skill.py` スクリプトはPythonとファイルシステムがあればどこでも動作する。Claude.aiでは実行でき、ユーザーは結果の `.skill` ファイルをダウンロードできる。

## Cowork固有の指示

Coworkにいる場合、主に知っておくべきこと:

- サブエージェントがあるので、メインワークフロー（テストケースの並列起動、ベースライン実行、採点など）はすべて動作する。（ただし、タイムアウトの深刻な問題に遭遇した場合は、テストプロンプトを並列ではなく直列で実行してもOK。）
- ブラウザやディスプレイがないため、評価ビューアーを生成する際は `--static <output_path>` でサーバーを起動する代わりにスタンドアロンHTMLファイルを書き出す。その後、ユーザーがブラウザで開けるリンクを提供する。
- 何らかの理由で、Coworkのセットアップではテスト実行後に評価ビューアーを生成することをClaudeが避ける傾向があるため、改めて強調する: CoworkでもClaude Codeでも、テスト実行後は必ず人間に例を見せるために評価ビューアーを生成すべきであり、`generate_review.py` を使用する（独自のブティックHTMLコードを書くのではない）。事前にお詫びするが、ここは全大文字にする: 自分で入力を評価して修正しようとする*前に*、評価ビューアーを生成する。できるだけ早く人間の前に出すべき！
- フィードバックの動作が異なる: 実行中のサーバーがないため、ビューアーの「Submit All Reviews」ボタンは `feedback.json` をファイルとしてダウンロードする。そこから読めばよい（最初にアクセスを要求する必要があるかもしれない）。
- パッケージングは動作する — `package_skill.py` はPythonとファイルシステムがあればOK。
- Description最適化（`run_loop.py` / `run_eval.py`）はCoworkでも問題なく動作するはず。ブラウザではなく `claude -p` をサブプロセスで使用するため。ただし、スキルが完全に完成し、ユーザーが良い状態だと同意してから行うこと。
